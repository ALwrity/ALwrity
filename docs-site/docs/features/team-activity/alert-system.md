# Alert System

ALwrity's agent alert system provides persistent, server-side notifications for serious findings from the ContentGuardianAgent, other agents, and the workflow engine. Alerts survive page reloads and are dismissible via API.

## Data Model

### `AgentAlert` Table

| Column | Type | Description |
|---|---|---|
| `id` | Integer, PK | Auto-increment |
| `user_id` | String(255) | Clerk user ID (indexed) |
| `source` | String(30) | Alert source, default `"agents"` |
| `alert_type` | String(50) | Category code, e.g. `guardian_agent_failing`, `guardian_coverage_gap` |
| `severity` | String(20) | `"info"`, `"warning"`, or `"error"` |
| `title` | String(255) | Human-readable alert title |
| `message` | Text | Detailed message |
| `cta_path` | String(255) | Optional call-to-action URL path |
| `payload` | JSON | Structured metadata (agent name, type, etc.) |
| `dedupe_key` | String(255) | Deduplication key (indexed) |
| `created_at` | DateTime | When the alert was created |
| `read_at` | DateTime, nullable | When the user dismissed it (null = unread) |

### Deduplication

Alerts are deduplicated by `dedupe_key`. The key format for guardian alerts is:

```
guardian:{alert_type}:{agent_name}:{title}
```

For example: `guardian:agent_failing:SEOOptimizationAgent:SEO Optimization needs attention`

If an alert with the same `dedupe_key` already exists, `create_alert()` is a no-op — preventing duplicate alerts across daily cycles.

### Unread Index

A composite index on `(user_id, read_at)` enables fast queries for unread alerts:

```python
Index("ix_agent_alerts_user_unread", AgentAlert.user_id, AgentAlert.read_at)
```

## API Endpoints

### GET `/api/agents/alerts`

Fetch alerts for the authenticated user.

| Parameter | Type | Default | Description |
|---|---|---|---|
| `unread_only` | boolean | `true` | Only return alerts where `read_at` is null |
| `limit` | integer | `50` | Maximum number of alerts |
| `detail_tier` | string | `"summary"` | Detail level: `"summary"`, `"standard"`, `"full"` |

### POST `/api/agents/alerts/{alert_id}/mark-read`

Mark an alert as read. Sets `read_at` to the current timestamp.

**Request:** No body required.

**Response:**

```json
{
  "success": true,
  "timestamp": "2026-05-31T10:00:00Z",
  "user_id": "user_..."
}
```

Returns `404` if the alert ID doesn't exist for this user.

## Alert Types

Alerts generated by the ContentGuardianAgent audit:

| `alert_type` | Severity | Trigger |
|---|---|---|
| `guardian_agent_failing` | error | Agent health score < 50 |
| `guardian_weak_reasoning` | warning | ≥3 weak reasoning proposals from one agent |
| `guardian_coverage_gap` | warning | A pillar received zero proposals |
| `guardian_proposal_overlap` | warning | Duplicate title proposed by multiple agents |

Other alert sources may use `alert_type` values like `system_check`, `agent_error`, etc.

## Frontend: AlertBanner

The `AlertBanner` component (`frontend/src/components/TeamActivity/AlertBanner.tsx`) renders alerts at the top of the Team Activity page.

### Display Logic

- Shows up to **5 unread alerts** ordered by severity (error → warning → info)
- Each alert row shows: severity icon, title, message, and a dismiss button
- Pending approvals are shown separately in a collapsible section

### Dismiss Behaviour

When a user clicks the dismiss (X) button:

1. The `dismissing` set is updated to show a loading state
2. A `POST /api/agents/alerts/{id}/mark-read` request is sent
3. On success: the alert ID is added to the local `dismissed` set
4. On API failure: the alert is still dismissed locally (graceful fallback)
5. The `dismissing` set is cleaned up in the `finally` block

```typescript
const handleDismiss = async (alertId: number) => {
  setDismissing((s) => new Set(s).add(alertId));
  try {
    await apiClient.post(`/api/agents/alerts/${alertId}/mark-read`);
    setDismissed((s) => new Set(s).add(alertId));
  } catch {
    setDismissed((s) => new Set(s).add(alertId)); // fallback
  } finally {
    setDismissing((s) => { const next = new Set(s); next.delete(alertId); return next; });
  }
};
```

## Backend Service

Alerts are created by the `AgentActivityService.create_alert()` method:

```python
def create_alert(
    self,
    alert_type: str,
    title: str,
    message: str,
    severity: str = "info",
    source: str = "agents",
    cta_path: str = None,
    payload: dict = None,
    dedupe_key: str = None,
) -> Optional[AgentAlert]:
```

- If `dedupe_key` is provided and an alert with the same key already exists, returns `None` (no-op)
- Otherwise, creates a new `AgentAlert` row with `read_at=None`
